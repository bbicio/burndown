# Test-branch.sh Port-Merge Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix `scripts/test-branch.sh`'s `write_override()` so the branch-specific Docker Compose override actually replaces each service's `ports` instead of Docker Compose concatenating it with the base `docker-compose.yml`'s ports — the bug that makes `scripts/test-branch.sh up` always fail with a port conflict whenever the main stack is already running.

**Architecture:** Add the Compose Specification's `!override` YAML merge tag to each of the four `ports` lists generated by `write_override()`'s heredoc (`db`, `api`, `nginx`, `adminer`). `!override` forces Compose to replace the field entirely on merge instead of its default list-concatenation behavior. No other function in the script changes; `docker-compose.yml` itself is untouched.

**Tech Stack:** Bash (`scripts/test-branch.sh`), Docker Compose Specification `!override` merge tag (supported since Compose v2.24; this repo's installed `docker compose version` is v5.1.4).

## Global Constraints

- No Docker Compose version guard/check added to the script (YAGNI — `!override` requires Compose ≥ v2.24, and this repo's installed version is v5.1.4; an unsupported version would surface its own clear Compose parse error).
- No change to `docker-compose.yml` — the main stack's behavior when run alone (`docker compose up`, no override file) must stay byte-identical to before this fix.
- No change to `container_name` handling in `write_override()` (already correct — Compose replaces scalars, not just list fields, on merge).
- No change to any other function in `scripts/test-branch.sh` (`load_env`, `wait_healthy`, `open_browser`, `status`, `up`, `down`) beyond the `write_override()` heredoc content itself.
- No change to `.claude/commands/finish-cycle.md`'s Gate 2 prose.

- [ ] **Step 2: Add the explanatory comment and `!override` tag**

In `scripts/test-branch.sh`, replace:

```bash
write_override() {
  cat > "$OVERRIDE_FILE" <<EOF
services:
  db:
    container_name: ${DB_CONTAINER}
    ports: ["${DB_PORT}:5432"]
  api:
    container_name: ${API_CONTAINER}
    ports: ["${API_PORT}:3000"]
  nginx:
    container_name: pdash-nginx-${SANITIZED}
    ports: ["${FRONTEND_PORT}:80"]
  adminer:
    container_name: pdash-adminer-${SANITIZED}
    ports: ["${ADMINER_PORT}:8080"]
EOF
}
```

with:

```bash
# Why !override on ports: Docker Compose concatenates list-type fields (like
# ports) across -f files instead of replacing them — container_name, a
# scalar, is replaced correctly, but a plain `ports: [...]` override would
# add to the base file's ports rather than replace them, causing this
# service to try to bind both the base port and the branch port. !override
# (Compose Specification merge tag, supported since Compose v2.24) forces
# a full replace instead.
write_override() {
  cat > "$OVERRIDE_FILE" <<EOF
services:
  db:
    container_name: ${DB_CONTAINER}
    ports: !override ["${DB_PORT}:5432"]
  api:
    container_name: ${API_CONTAINER}
    ports: !override ["${API_PORT}:3000"]
  nginx:
    container_name: pdash-nginx-${SANITIZED}
    ports: !override ["${FRONTEND_PORT}:80"]
  adminer:
    container_name: pdash-adminer-${SANITIZED}
    ports: !override ["${ADMINER_PORT}:8080"]
EOF
}
```
