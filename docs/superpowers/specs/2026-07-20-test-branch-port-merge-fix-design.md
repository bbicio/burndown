# Design — Fix port-merge bug in `scripts/test-branch.sh`

**Date:** 2026-07-20
**Brief:** `docs/superpowers/briefs/2026-07-20-test-branch-port-merge-bug-brief.md`
**Scenario:** 2 — Evolution (bug fix on existing behavior)

## Problem

`write_override()` (`scripts/test-branch.sh:62-78`) generates `docker-compose.branch.yml` with branch-specific `ports` values for `db`/`api`/`nginx`/`adminer`. Docker Compose concatenates `ports` lists across multiple `-f` files instead of replacing them (`container_name`, a scalar, is correctly replaced — only list-type fields like `ports` have this behavior). Verified via `docker compose -p <project> -f docker-compose.yml -f docker-compose.branch.yml config`: the resolved `db` service published both `5432` (from `docker-compose.yml:14-15`) and `5433` (from the override), and the same pattern was confirmed for `adminer`. `scripts/test-branch.sh up` therefore always fails with "port is already allocated" whenever the main stack (or anything else on those base ports) is already running — the tool's stated primary use case.

## Fix

Add the Compose Specification's `!override` YAML merge tag to each of the four `ports` lists in `write_override()`'s heredoc. `!override` tells Compose to replace the field entirely instead of concatenating it — verified empirically (`docker compose ... config` with a test override file) that this produces a resolved `db` service publishing only `5433`, not both `5432` and `5433`.

```bash
# Why !override on ports: Docker Compose concatenates list-type fields (like
# ports) across -f files instead of replacing them — container_name, a
# scalar, is replaced correctly, but a plain `ports: [...]` override would
# add to the base file's ports rather than replace them, causing this
# service to try to bind both the base port and the branch port. !override
# (Compose Specification merge tag, supported since Compose v2.24) forces
# a full replace instead.
write_override() {
  cat > "$OVERRIDE_FILE" <<EOF
services:
  db:
    container_name: ${DB_CONTAINER}
    ports: !override ["${DB_PORT}:5432"]
  api:
    container_name: ${API_CONTAINER}
    ports: !override ["${API_PORT}:3000"]
  nginx:
    container_name: pdash-nginx-${SANITIZED}
    ports: !override ["${FRONTEND_PORT}:80"]
  adminer:
    container_name: pdash-adminer-${SANITIZED}
    ports: !override ["${ADMINER_PORT}:8080"]
EOF
}
```

All four services' `ports` get `!override`, not just `db` — the same concatenation bug applies to `api`, `nginx`, and `adminer` (all four have distinct base vs. branch ports in `docker-compose.yml` vs. the generated override), even though only `db` and `adminer` were empirically checked so far; the mechanism is identical Compose behavior for any list-type field, so no separate per-service verification is needed to justify applying the same fix uniformly.

No other line in `scripts/test-branch.sh` changes. No changes to `docker-compose.yml`.

## Explicitly not done

- No Docker Compose version guard/check added to the script. `!override` requires Compose ≥ v2.24 (January 2024); given this repo's own installed version is v5.1.4 and the constraint against unnecessary complexity, a version check is not warranted (YAGNI) — if it ever becomes an issue, Compose itself reports a clear parse error for an unsupported merge tag.
- No change to `container_name` handling (already correct — scalars are replaced, not concatenated, by Compose's own merge rules).

## Testing

Manual verification (no automated shell-test framework in this repo, consistent with prior cycles touching this script):

1. `docker compose -p pdash_branch_<test> -f docker-compose.yml -f docker-compose.branch.yml config` (with a `docker-compose.branch.yml` generated by the fixed `write_override()`) shows each of `db`/`api`/`nginx`/`adminer` publishing **only** its branch-specific port — no base port present in the resolved config.
2. With the main stack running (`docker compose up`, unmodified `docker-compose.yml`, base ports 5432/3000/80/8080 occupied), `scripts/test-branch.sh up` on a feature branch succeeds without a port conflict.
3. After step 2, `scripts/test-branch.sh down` tears down cleanly.
4. The main stack itself, run alone via `docker compose up` with no override file present, still publishes 5432/3000/80/8080 exactly as before this fix (config unaffected, since `docker-compose.yml` itself is untouched).
